📏 1. CHUNK_SIZE

Ukuran potongan teks (dalam jumlah kata) saat dokumen dipecah menjadi bagian-bagian kecil.

# Buka app.py, cari baris ini:
CHUNK_SIZE = 300 # ubah angka ini
Tekan ini untuk lihat jawaban

Apa itu CHUNK_SIZE?

Saat dokumen PDF diupload, teksnya dipotong-potong menjadi bagian kecil (chunk) agar bisa diproses dan dicari. CHUNK_SIZE menentukan berapa banyak kata per potongan.

Pengaruhnya:

  • Kecil (100-200) — Potongan lebih presisi, cocok untuk pertanyaan spesifik. Tapi konteks bisa terlalu sempit dan jawaban kurang komprehensif.
  • Sedang (250-400) — Keseimbangan antara presisi dan kelengkapan konteks. Recommended.
  • Besar (500-1000) — Konteks lebih kaya, tapi pencarian kurang presisi dan bisa mengambil informasi yang tidak relevan.
💡 Jika jawaban terasa "terpotong" atau kurang konteks, naikkan CHUNK_SIZE. Jika jawaban terlalu umum, turunkan.

🔍 2. TOP_K_INITIAL

Jumlah kandidat awal yang diambil dari vector search sebelum di-filter/rerank.

TOP_K_INITIAL = 20
Tekan ini untuk lihat jawaban

Apa itu TOP_K_INITIAL?

Setelah pertanyaan diubah jadi vektor, sistem mencari chunk yang paling mirip. TOP_K_INITIAL menentukan berapa banyak kandidat awal yang diambil dari hasil pencarian vektor.

Pengaruhnya:

  • Kecil (5-10) — Lebih cepat, tapi bisa melewatkan chunk relevan yang berada di urutan bawah.
  • Sedang (15-25) — Cukup untuk menangkap kandidat relevan tanpa terlalu lambat. Recommended.
  • Besar (30-50) — Recall lebih tinggi (lebih banyak kemungkinan relevan), tapi proses reranking jadi lebih lambat.
💡 Nilai ini hanya berpengaruh signifikan jika USE_RERANKER = True. Tanpa reranker, yang dipakai langsung hanya FINAL_TOP_M teratas.

📋 3. FINAL_TOP_M

Jumlah konteks final yang diberikan ke LLM untuk menghasilkan jawaban.

FINAL_TOP_M = 5
Tekan ini untuk lihat jawaban

Apa itu FINAL_TOP_M?

Dari sekian banyak kandidat yang ditemukan, hanya M teratas (setelah reranking) yang dikirim ke LLM sebagai konteks untuk menjawab pertanyaan.

Pengaruhnya:

  • Sedikit (2-3) — Jawaban lebih fokus dan cepat. Risiko: konteks penting bisa terlewat.
  • Sedang (4-6) — Keseimbangan baik antara kelengkapan dan fokus. Recommended.
  • Banyak (7-10) — Jawaban lebih komprehensif tapi bisa membingungkan LLM jika konteks saling bertentangan, dan lebih lambat/mahal.
💡 Semakin banyak konteks = semakin banyak token yang dikirim ke LLM = latency lebih tinggi. Jaga agar tetap efisien.

🎯 4. USE_RERANKER

Aktifkan/nonaktifkan Cross-Encoder reranker untuk memperbaiki urutan hasil pencarian.

USE_RERANKER = True # atau False
Tekan ini untuk lihat jawaban

Apa itu Reranker?

Reranker adalah model tambahan (Cross-Encoder) yang mengevaluasi ulang pasangan [pertanyaan, chunk] untuk mengurutkan relevansi lebih akurat daripada hanya cosine similarity biasa.

Pengaruhnya:

  • True — Presisi naik signifikan (bisa 10-30% lebih akurat). Konteks yang dikirim ke LLM lebih relevan. Tapi ada tambahan latency ~100-300ms.
  • False — Lebih cepat, tapi urutan konteks hanya berdasarkan cosine similarity yang kadang kurang akurat.
💡 Untuk dokumen teknis atau pertanyaan yang ambigu, reranker sangat membantu. Untuk kecepatan maksimal, matikan.

🌡️ 5. TEMPERATURE

Mengontrol seberapa "kreatif" vs "deterministik" jawaban dari LLM.

TEMPERATURE = 0.2 # range: 0.0 - 1.0
Tekan ini untuk lihat jawaban

Apa itu Temperature?

Temperature mengatur distribusi probabilitas saat LLM memilih kata berikutnya. Semakin rendah, semakin "pasti" dan konsisten jawabannya.

Pengaruhnya:

  • 0.0 - 0.2 — Jawaban sangat konsisten, deterministik, faktual. Ideal untuk QA berbasis dokumen. Recommended untuk RAG.
  • 0.3 - 0.5 — Sedikit variasi, masih cukup akurat. Cocok untuk ringkasan.
  • 0.6 - 0.8 — Lebih kreatif dan bervariasi. Cocok untuk brainstorming.
  • 0.9 - 1.0 — Sangat acak/kreatif. Bisa menghasilkan jawaban yang tidak akurat atau "halusinasi".
⚠️ Untuk RAG (jawaban berdasarkan dokumen), gunakan temperature rendah (0.1-0.3) agar jawaban tetap grounded pada konteks yang diberikan.

🤖 6. GROQ_MODEL

Model LLM yang digunakan untuk menghasilkan jawaban melalui Groq API.

GROQ_MODEL = "llama-3.1-8b-instant" # model default

Contoh model lain yang bisa dipakai:

# Ganti value GROQ_MODEL ke salah satu di bawah:

GROQ_MODEL = "openai/gpt-oss-120b"
GROQ_MODEL = "llama-3.3-70b-versatile"
Tekan ini untuk lihat jawaban

Apa bedanya tiap model?

Setiap model punya ukuran parameter dan kemampuan yang berbeda. Secara umum:

llama-3.1-8b-instant (Default — 8B parameter)

  • Paling cepat, latency rendah (~200-500ms)
  • Jawaban cukup baik untuk pertanyaan sederhana
  • Bisa kesulitan dengan reasoning kompleks atau multi-hop
  • Cocok untuk: prototipe, demo, penggunaan sehari-hari

llama-3.3-70b-versatile (70B parameter)

  • Jauh lebih pintar dan akurat, reasoning kuat
  • Latency lebih tinggi (~1-3 detik)
  • Jawaban lebih detail, nuanced, dan komprehensif
  • Cocok untuk: analisis dokumen kompleks, pertanyaan teknis

openai/gpt-oss-120b (120B parameter)

  • Model open-source raksasa dengan kapasitas memori luar biasa
  • Akurasi sangat tinggi untuk tugas reasoning yang sangat sulit
  • Latency paling tinggi di antara pilihan lain
  • Cocok untuk: pengolahan dokumen sangat panjang atau logika rumit
💡 Tips: Mulai dengan llama-3.1-8b-instant untuk development. Saat butuh akurasi tinggi, switch ke llama-3.3-70b-versatile atau openai/gpt-oss-120b.

⚙️ Parameter Tambahan (Non-Tunable)

Parameter ini sudah di-set optimal. Ubah hanya jika paham dampaknya.

CHUNK_OVERLAP = 60 # overlap antar chunk (kata)
SCORE_THRESHOLD = 0.30 # skor minimum cosine similarity
Tekan ini untuk lihat jawaban

CHUNK_OVERLAP (60) — Jumlah kata yang tumpang tindih antar chunk. Ini memastikan informasi di perbatasan chunk tidak terpotong. Terlalu besar = banyak duplikasi, terlalu kecil = konteks bisa hilang di batas potongan.

SCORE_THRESHOLD (0.30) — Skor minimum agar sebuah chunk dianggap relevan. Chunk dengan cosine similarity di bawah 0.30 akan dibuang. Turunkan jika terlalu banyak pertanyaan mendapat "tidak ditemukan", naikkan jika jawaban terlalu banyak noise.
